# 产品需求文档 (PRD)：AI 赋能企业供应链可视化分析系统 (MVP 版)

## 1. 文档概述

本文档旨在定义"AI 赋能企业供应链可视化分析系统"MVP 版本的功能需求与非功能需求，作为后续系统设计、开发与验收的依据。

- **项目名称：** AI 赋能企业供应链可视化分析系统
- **文档版本：** V1.0
- **适用阶段：** MVP（最小可行产品）开发阶段
- **开发周期：** 1 个月
- **目标读者：** 产品经理、开发工程师、测试工程师、项目评审人员

## 2. 软件系统的一般性描述

### 2.1 系统定位

本项目为一套轻量级的供应链可视化与智能预警系统。鉴于开发周期为一个月，本阶段不直接对接企业真实业务系统，而是采用 Kaggle 开源供应链数据集（DataCoSupplyChainDataset），通过构建"数据回放引擎"模拟实时高并发数据流，结合 AI 算法实现对供应链链路状态的动态监控与异常预警。

### 2.2 核心目标

- **数据流动可见：** 实现静态数据集的时序化回放，前端大屏能实时展示动态刷新的订单流与物流状态。
- **智能风险阻断：** 建立基于机器学习的风险预警模型，毫秒级识别异常订单并主动推送告警。
- **闭环演示交付：** 形成"数据产生 → 后端接收 → AI 判定 → 前端预警"的完整闭环，支撑最终的项目演示与答辩。

### 2.3 目标用户

- **主要角色：** 供应链风控专员 / 管理层
- **核心场景：** 用户打开系统大屏，看到当天的核心指标（总单量、准交率）在实时跳动。突然，系统底层拦截到一条高风险的模拟订单数据，大屏右侧的"风险事件追踪"列表顶部立即闪烁红色高亮，提示某笔重要物资存在"极高延迟交付风险"，并展示预测依据。

## 3. 软件功能需求

### 3.1 软件功能概述

| 模块       | 功能概述                                                   |

#### （2）用例的交互模型及描述

**① UC-01 启动数据回放**

交互模型（时序）：

```
操作员 ──→ [数据回放引擎] ──→ [后端 API: POST /api/v1/stream/ingest] ──→ [MySQL]
```

自然语言描述：

1. 操作员在终端执行启动命令 `python replay_script.py`。
2. 数据回放引擎读取预处理后的 CSV 数据文件，按配置的倍率（如 1 秒 = 1 小时）逐条解析订单记录。
3. 引擎将每条记录封装为 JSON 格式，通过 HTTP POST 请求发送至后端 `/api/v1/stream/ingest` 接口。
4. 后端接口接收数据后，将结构化数据持久化存入 MySQL 数据库，并触发后续的 AI 推理流程。
5. 引擎持续运行直至全部数据回放完毕或操作员手动终止。

**② UC-02 实时风险识别**

交互模型（时序）：

```
[后端 API] ──→ [模型推理模块] ──→ [MySQL] ──→ [Redis]
```

自然语言描述：

1. 后端 API 在成功接收订单数据后，将该订单记录转发至模型推理模块（model_runner）。
2. 模型推理模块加载预先训练好的 `risk_model.pkl` 文件，提取订单特征进行推理。
3. 推理输出该订单的风险概率（0-1 之间的浮点数）。
4. 将风险评分结果写入 MySQL 订单记录，同时将评分缓存至 Redis。
5. 若风险概率超过 85%，则进入 UC-03 告警推送流程。

**③ UC-03 风险告警推送**

交互模型（时序）：

```
[模型推理模块] ──→ [WebSocket 服务] ──→ [前端大屏]
```

自然语言描述：

1. 当模型推理模块判定某订单风险概率 > 85% 时，构造告警 JSON 对象（含订单号、风险类型、风险概率、时间戳）。
2. 后端通过 WebSocket 通道 `ws://domain/api/v1/ws/alerts` 广播该告警消息。
3. 所有连接中的前端客户端实时接收告警消息。
4. 前端解析告警信息，更新预警列表并在大屏上以红色高亮闪烁展示。

**④ UC-04 全局 KPI 监控**

交互模型（时序）：

```
[风控专员] ──→ [前端大屏] ──WebSocket──→ [后端] ──→ [Redis/MySQL]
```

自然语言描述：

1. 风控专员打开浏览器访问大屏首页。
2. 前端与后端建立 WebSocket 长连接。
3. 后端根据 Redis 中缓存的实时聚合数据，定期通过 WebSocket 向前端推送更新的 KPI 值（累计订单量、GMV、风险拦截次数）。
4. 前端接收到推送后，动态更新页面上的数字显示区域，实现数值跳动效果。

**⑤ UC-05 物流热力图查看**

交互模型（时序）：

```
[风控专员] ──→ [前端大屏] ──HTTP REST──→ [后端 API] ──→ [MySQL]
```

自然语言描述：

1. 风控专员在大屏上切换到物流热力图视图。
2. 前端通过 HTTP GET 请求获取当前在途订单的物流节点与路由数据。
3. 后端从 MySQL 查询最新物流状态数据并返回。
4. 前端使用 ECharts 的地图组件渲染全国/全球物流路线，以动态光点和连线表示运输中的货物。

**⑥ UC-06 预警列表查看**

交互模型（时序）：

```
[风控专员] ──→ [前端大屏] ──WebSocket──→ [后端]
```

自然语言描述：

1. 风控专员在大屏右侧查看"风险事件追踪"列表区域。
2. 前端通过 WebSocket 持续接收后端的告警推送（来自 UC-03）。
3. 新告警到达时，前端将其插入列表顶部并高亮闪烁。
4. 列表条目展示单号、风险类型（延迟交付/欺诈）、风险概率和拦截时间。
5. 列表自动滚动，保持最新的高风险条目可见。

**⑦ UC-07 历史趋势查看**

交互模型（时序）：

```
[风控专员] ──→ [前端大屏] ──HTTP REST──→ [后端 API] ──→ [MySQL]
```

自然语言描述：

1. 风控专员在大屏上查看历史趋势折线图区域。
2. 前端通过 HTTP GET 请求获取指定时间范围（如过去 24 小时）的进出货量统计数据。
3. 后端从 MySQL 聚合查询按时段分组的订单量数据并返回。
4. 前端使用 ECharts 折线图展示进出货量随时间波动的趋势。

#### （3）软件需求的分析类模型及描述

基于以上用例，抽象出以下核心分析类：

| 类名                           | 类型  | 属性                                                                                                                                                                                       | 职责                                              |
| ---------------------------- | --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| Order（订单）                    | 实体类 | order_id, order_date, shipping_mode, delivery_status, customer_city, customer_country, latitude, longitude, product_category, order_amount, profit_ratio, risk_score, late_delivery_risk | 表示一条供应链订单记录，包含物流信息与 AI 推理结果                     |
| Alert（告警）                    | 实体类 | alert_id, order_id, risk_type, risk_probability, created_at, status                                                                                                                      | 表示一条由 AI 触发的风险告警，关联到具体订单                        |
| KPI（指标）                      | 值对象 | total_orders, total_gmv, risk_intercept_count, on_time_delivery_rate                                                                                                                     | 封装大屏展示的核心业务指标数据                                 |
| ReplayEngine（数据回放引擎）         | 控制类 | source_file, replay_rate, batch_size                                                                                                                                                     | 负责从 CSV 读取数据并按倍率注入后端                            |
| RiskModel（风险模型）              | 控制类 | model_path, threshold                                                                                                                                                                    | 负责加载训练好的模型文件、对输入特征进行推理、输出风险概率                   |
| IngestController（注入控制器）      | 边界类 | —                                                                                                                                                                                        | 接收 HTTP POST 请求的订单数据，协调 Order 持久化与 RiskModel 推理 |
| AlertPushController（告警推送控制器） | 边界类 | —                                                                                                                                                                                        | 管理 WebSocket 连接，将高风险的 Alert 广播至前端               |
| DashboardView（大屏视图）          | 边界类 | —                                                                                                                                                                                        | 前端大屏展示层，整合 KPI、地图、列表、折线图等子视图                    |

类关系描述：

- ReplayEngine 通过 HTTP 调用 IngestController，IngestController 创建 Order 对象并委托 RiskModel 进行风险推理。
- RiskModel 根据推理结果，若风险超过阈值则创建 Alert 对象。
- AlertPushController 监听 Alert 的创建事件，通过 WebSocket 推送给 DashboardView。
- DashboardView 聚合展示 KPI、Order 物流信息、Alert 列表和历史趋势。

#### （4）对象的状态模型及描述

**① Order（订单）对象的状态模型**

```
                    ┌──────────┐
                    │  已创建   │
                    │ (Created) │
                    └─────┬────┘
                          │
                    ┌─────▼────┐
                    │  已注入   │
                    │(Ingested) │
                    └─────┬────┘
                          │
                    ┌─────▼────┐
                    │  已分析   │
                    │(Analyzed) │
                    └──┬───┬───┘
                       │   │
              ┌────────┘   └────────┐
              ▼                      ▼
     ┌────────────┐        ┌────────────────┐
     │  正常流转   │        │  高风险待处理   │
     │  (Normal)  │        │ (HighRisk)     │
     └────────────┘        └───────┬────────┘
                                   │
                          ┌────────▼────────┐
                          │    已拦截阻断    │
                          │   (Blocked)     │
                          └─────────────────┘
```

自然语言描述：

- **已创建 (Created)：** 订单从 CSV 数据源被读取并解析为 Order 对象。
- **已注入 (Ingested)：** 订单通过 HTTP 接口成功注入后端系统，已持久化至 MySQL。
- **已分析 (Analyzed)：** AI 模型完成对该订单的风险推理，生成风险评分。
- **正常流转 (Normal)：** 风险评分低于阈值，订单正常进入后续物流流程。
- **高风险待处理 (HighRisk)：** 风险评分超过 85% 阈值，触发告警生成。
- **已拦截阻断 (Blocked)：** 系统自动标记该订单为已拦截，防止异常订单继续流转。

**② Alert（告警）对象的状态模型**

```
     ┌──────────┐
     │  已生成   │
     │(Created) │
     └─────┬────┘
           │
     ┌─────▼────┐
     │  已推送   │
     │ (Pushed) │
     └─────┬────┘
           │
     ┌─────▼────┐
     │  已确认   │
     │(Acknowledged)│
     └──────────┘
```

自然语言描述：

- **已生成 (Created)：** 当订单的风险评分超过阈值时，系统自动创建 Alert 对象并关联到该订单。
- **已推送 (Pushed)：** Alert 已通过 WebSocket 通道成功广播至所有在线前端客户端。
- **已确认 (Acknowledged)：** 风控专员在前端点击确认该告警，表明已对该风险事件进行人工复核。
